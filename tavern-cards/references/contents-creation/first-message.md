# 开场白创作

开场白是故事的开始，设定初始场景和情境，体现角色性格、设定场景基调、引导用户进入互动。仅角色卡需要此步骤。

编写前读取编写规划文档的 `first_message` 段落（format/word_count/scene/opening_situation），按其指引创作。叙事式时同时读取 `style` 段落（perspective/tone/mood/reference）。

## MVU 变量确认

项目使用了 MVU 时，先读取 initvar.yaml，确保开场白的初始状态与变量一致。

## 两种格式

### 叙事式

传统的角色扮演开场白，AI 直接叙述场景，以角色身份说话。适合大多数情况。

编写要点：
- 从场景描写开始，逐步聚焦到角色
- 角色通过行为和对话自然出场，不做人物介绍式的罗列
- 结尾留出互动点——让 <user> 有明确的事可回应
- 不要替 <user> 做决定、说话、或预设 <user> 的行动
- 如果 MVU，确保叙述中的状态与 initvar.yaml 一致

错误示范：
- 人物介绍式："你叫小明，今年17岁，是..."（不要预设 <user>）
- 封闭式结尾："于是你们幸福地生活在一起了"（没有互动空间）
- 变量不一致：initvar 写好感度=0，开场白却写"你们已是多年老友"

示例（正例——有钩子 + 身份动机明确 + 结尾有方向）：

```
面试通过的通知只有一行字："下周一上午十点，排练室见。"

附件的压缩包里没有合同，只有两首未完成的demo和一个地址——巷子尽头、没有店招、地下二层。你问了三个熟人才确认那是家真的Livehouse而不是什么别的地方。

现在你站在那扇掉了漆的铁门前。门缝里漏出一段吉他声，同一段旋律反复地弹，每次都断在同一个尾音上。

你推开门。

排练室比想象中亮。午后的阳光从靠近天花板的小窗户斜进来，在木地板上画出一块块金黄。秋明月坐在音箱上，手指漫不经心地拨着吉他弦——她就是那段旋律的源头。她听见门响，手指停下来，但没有抬头。

"来了？"

她把拨片在指间转了一圈，又继续弹那几个音。没抬头，语气平淡——但面前摆了一把空椅子，搁着一杯温过的水。

排练室的时钟指向十点过三分。门外有人搬动设备的声响。那截断掉的旋律还在等你接上它。
```

反例——太平淡：

```
秋日的午后，阳光很好。你走在街上，准备去排练室。你推开门，看见秋明月坐在那里。

她看见你来了，说了一声"来了？"。
```

### 大纲式

只整理关键信息的大纲，不写完整描写。适合用户有清晰的场景构想、希望精确控制内容的情况。

编写要点：
- 只根据用户提供的信息整理，不自己编造内容
- 不擅自填充用户没说的细节
- 结构清晰，关键要素一目了然

格式参考：

```
场景: [时间/地点/情境一句话]
背景设定:
  - [背景信息1]
  - [背景信息2]
当前状态:
  [角色A]状态:
    - [状态描述]
  [角色B]状态:
    - [状态描述]
关键细节:
  - [具体细节1]
  - [具体细节2]
开场点:
  1. [互动切入点1]
  2. [互动切入点2]
```

## 开场吸引力

开场白必须在开头几行抓住玩家注意力，否则玩家可能不会继续。无论哪种格式，开头几句决定玩家是否愿意投入。

编写要点：
- 开篇制造悬念或情绪拉力——一个未解决的问题、一个反常的细节、一个让玩家想问"为什么"的瞬间
- 不要从平淡的日常流水账开始（"今天天气很好，你走在街上..."）
- 用具体的细节制造画面感和独特感，避免泛泛的都市描写
- 如果项目有异能/超自然设定，可以暗示隐藏层，不必一开始就全盘托出

错误示范（太平淡）：
- "你走在一条普通的街道上，阳光很好，你准备去上班"
- "这是一个普通的周三下午，你刚吃完午饭"

正例示范（有拉力）：
- "聘用信上只有三行字，没有公司名，没有岗位说明，只写了一个地址和一个时间。落款是一个你不认识的邮箱。"
- "你第一次站在 Pink Stage 门口的时候，并没有想到推开这扇门之后的生活会变成什么样。"

## 剧情动力

玩家阅读开场白后必须知道：
1. **我是谁**——玩家的身份和角色定位（新来的运营？制作人？推荐来的朋友？）
2. **我为什么在这里**——来到这个场景的合理动机（应聘？被委托？收到邀请？）
3. **我能做什么**——开场结尾留给玩家一个或多个清晰的行为方向

编写要点：
- 在开场中交代玩家的岗位/身份来源，而不只是"你到了"（比如收到招聘信息的过程、被熟人推荐、通过某个面试等）
- 结尾给出 2-3 个自然的选择方向，让玩家可以立刻决定下一步行动
- 如果项目有异能/超自然设定，剧情动力中应暗示（而非明示）这层隐藏线，让玩家有探索欲望
- "后续会自然展开"式被动等待不够——玩家需要知道自己该主动做什么

错误示范：
- 仅设定场景却未交代玩家的角色/动机
- 结尾是封闭的叙述，没有给玩家选择的空间

正例示范（结尾选择方向）：
- "走廊尽头的吉他声又断在了同一个地方。你可以去看看星野萤那边需不需要帮忙，也可以先回后台找紫藤泉拿资料。手机又震了一下——朝仓奈奈刚发了一条消息："我到啦！顺便买了泡芙！>_<""

## 额外开场白

额外开场白需要与默认 initvar 不同的变量初始值时，先读取 `references/mvu/initvar.md` 了解编写原则，然后在消息内容末尾嵌入 `<UpdateVariable><initvar>...</initvar></UpdateVariable>`：

```
<UpdateVariable>
<initvar>
{initvar 内容}
</initvar>
</UpdateVariable>
```

**约束**：额外开场白的 `<initvar>` 块完全覆盖世界书的默认 initvar，不是合并。不需要不同的变量时不加这个块。

## 产出

保存到项目目录下的 `开场白/{index}.txt`（0.txt 为第一条，额外开场白为 1.txt、2.txt 等）。

保存后注册到 state 文件：

```bash
node scripts/tavern-cards-forge.mjs patch {project} '[{"op": "add", "path": "/first_messages/-", "value": "开场白/{index}.txt"}]'
```

额外开场白需在消息内容末尾嵌入 `<UpdateVariable><initvar>` 块设置对应的初始变量值。

叙事式开场白的描写只需遵守 `references/rules.md` 中的八股化检查，其余规则不适用于叙事文本。

## 自查清单

- [ ] 开头 3 行内制造吸引力——有悬念、反常细节或情绪拉力
- [ ] 剧情动力完整：玩家身份明确、到场动机清晰、结尾有行为方向
- [ ] 叙事式：角色通过行为和对话出场，不是人物介绍式罗列
- [ ] 叙事式：结尾留出了互动点，<user> 有明确的事可回应
- [ ] 叙事式：没有替 <user> 做决定、说话、或预设行动
- [ ] 叙事式：检查字数是否符合要求
- [ ] 大纲式：只根据用户提供的信息整理，没有自己编造内容
- [ ] MVU 项目：叙述中的状态与 initvar.yaml 一致
- [ ] 额外开场白：需要不同变量值时已嵌入 `<UpdateVariable><initvar>` 块
