# 从材料转化

当用户有现成素材（轻小说、游戏脚本、小说等）时，需求对齐阶段的世界信息和角色信息不通过提问收集，改为从源材料提取。创作流程与从零创作完全相同。

## 流程

### 前置：评估材料类型

通读所有源材料后，判断结构类型，选择对应路径：

- **结构化材料**（YAML、JSON、结构化文档、纯数据文件）：信息已按角色/世界等分类组织，无需章节级索引
  → 跳过"叙事材料路径"，直接进入"生成条目规划"
- **叙事/松散材料**（小说、游戏脚本、散文式笔记、故事文本、Markdown 长篇）：信息散落在叙述中，需要章节级索引来追踪
  → 按下方"叙事材料路径"：初读构建大纲 → 整理深化

两种路径在此分支，最终汇聚到"生成条目规划"。

---

### 叙事材料路径

信息散落在叙述中时，用大纲提取章节级索引，确保提取不遗漏。

#### 第一步：初读构建大纲

通读源材料，边读边将信息记录到 `故事大纲.yaml`（章节级结构，后续通过 `project-setup.md` 步骤 6 移入项目目录）：

```yaml
chapters:
  - name: 第1章
    lines: L1-L350
    worldview: [规则/背景设定...]
    characters: [角色首次出场/性格体现...]
    items: [道具/能力首次出现...]
    style: [叙事手法...]
    important: [关键描写: 外貌细节/关系转折/能力首现...]
```

章节的行号索引是后续精准复读的依据，务必准确。

#### 第二步：整理深化

通读完成后，在第一步构建的章节级结构基础上，补充全局性信息。

```yaml
chapters:
  - name: 第1章
    lines: L1-L350
    ...

worldview_summary: |
  世界观总纲（100-200字）

characters_summary:
  - name: 林小雨
    summary: 一句话定位

items_summary:
  - name: xxx
    summary: 一句话描述

story_summary: |
  故事/章节总纲

important_chapters:
  - chapter: 第5章
    reason: 林小雨外貌首次详细描写
```

故事大纲是编写条目时的参考索引，不直接成为世界书条目。

---

### 生成条目规划

根据结构化材料或故事大纲，生成条目规划（写入编写规划文档，见 `references/requirements.md`）。

**结构化材料路径**：直接遍历每个源文件，按文件内容的结构层级提取信息填入条目规划。例如 YAML 文件的顶层字段（角色、世界观等）直接映射为对应类型的条目，无需 `source_chapters`。

**叙事材料路径**：根据故事大纲逐章提取，每个条目额外标注源材料依赖：

```yaml
entries:
  - name: 林小雨_基础信息
    type: 角色
    part: basic
    path: 世界书/角色/林小雨/基础信息.yaml
    source_chapters: [第1章, 第5章]  # 编写此条目时需要复读的章节
```

`source_chapters` 列出编写该条目时需要参考的源材料章节，用于精准复读——只读这些章节的行号范围，不重读全文。

以下场景必须标注 source_chapters：
1. 角色外貌在不同章节有变化
2. 关系转折点在特定章节
3. 能力首次使用的具体效果

### 进入创作流程

用户确认条目规划后，进入与从零创作相同的流程（见 SKILL.md 完整项目流程）。

编写每个条目时，结构化材料无需复读，直接按创作规则编写。叙事材料先根据 source_chapters 精准复读对应章节，再按创作规则编写。

## 转化特有约束

- **去重**：多个章节描述同一角色 → 合并为一个条目
- **不编造**：原文没说的不自行补全
- **文学→数据**：原文散文转为 YAML 数据库格式
- **信任距离**：每个提取特征必须有原文证据
- **关键信息确认**：源材料中缺失的关键具体信息（如城市名、学校名、组织名等），整理为问题列表在转化前一次性询问用户；不得自行捏造或用模糊词替代
